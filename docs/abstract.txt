STRIDE: AI-Agent-Driven Automated Test Execution and Root-Cause Analysis Pipeline

Abstract

Validation of distributed system security and administration subsystems requires executing multi-step interactive command sequences where runtime outputs from one step feed into subsequent steps as parameters. The existing workflow requires engineers to manually transfer scripts via SCP, execute commands over SSH, retrieve logs, upload them to an AI assistant for analysis, and correlate failures across steps — a 12-step process consuming 30-60 minutes and 50,000-100,000 LLM tokens per test cycle. This paper presents STRIDE, a Python-based pipeline that automates the complete test lifecycle from specification to diagnosis using a single CLI command. The framework parses Excel test cases into YAML specifications with dynamic variable extraction patterns, executes multi-step commands over SSH with crash-resilient isolation, performs deterministic line-level failure detection via pattern matching, and invokes AI agents only for final report generation using pre-analyzed structured summaries rather than raw logs. The architecture achieves a 97% reduction in token consumption by ensuring that raw log data never reaches the LLM — only distilled failure metadata (~1,500 tokens vs. ~80,000 tokens per cycle) is forwarded for human-readable report generation. Validation on two production test suites (5 multi-step tests with 35 total steps, and 30 single-step tests) demonstrates complete unattended execution with accurate root-cause classification in under 5 minutes per cycle.

Problem Statement

Modern distributed systems provide mission-critical infrastructure for global financial services, telecommunications and healthcare institutions that require continuous availability. These systems run multiple security and administration subsystems that require rigorous validation before each release.

Testing these subsystems presents unique challenges. Commands must be executed interactively through nested shell environments — each layer invoking the next, with subsystem-specific tools accessible only through specific parent shells. Test sequences are inherently multi-step: a cryptographic key or resource must first be generated, its runtime-assigned identifier extracted from the output, and that identifier supplied to subsequent commands for verification, rotation or deletion. This creates data dependencies between steps that cannot be resolved statically.

The current workflow to validate a single test case involves:
(1) Writing test commands in Excel, (2) feeding them to an AI assistant to generate a script, (3) reviewing and fixing the script, (4) transferring the script to the remote node via SCP, (5) SSH login and execution, (6) collecting output, (7) transferring logs back to local machine, (8) uploading logs to an AI assistant, (9) prompting for failure identification, (10) manually correlating across steps, (11) diagnosing root cause, (12) writing findings into a report.

This process is fundamentally inefficient for three reasons. First, the 12 human-driven steps take 30-60 minutes per cycle and do not scale with increasing test count. Second, the AI-assisted analysis requires uploading full raw logs (500-2000 lines per test, totaling 50,000-100,000 input tokens for a 30-test suite) every cycle — re-analyzing identical patterns from scratch each time at significant token cost. Third, the workflow is non-deterministic: different engineers may upload different context windows, phrase prompts differently, and receive varying diagnoses for the same failure, making results unreproducible.

Our Solution

System Architecture

The proposed solution is implemented as a Python-based test execution and analysis framework. The end-to-end pipeline covers four stages: test execution (test_runner.py), log collection, deterministic analysis, and report generation (pipeline.py).

```
┌─────────────┐     ┌───────────────┐     ┌─────────────────┐
│   Excel     │────→│ excel_to_yaml │────→│  tests/*.yaml   │
│  Test Cases │     │  (converter)  │     │  (executable)   │
└─────────────┘     └───────────────┘     └────────┬────────┘
                                                    │
                    ┌───────────────────────────────┼───────────────────┐
                    │              test_runner.py    ↓                   │
                    │  ┌──────────┐  ┌───────────────┐  ┌───────────┐  │
                    │  │   SSH    │→ │  Step-by-Step │→ │  Variable │  │
                    │  │ Connect  │  │  Execution    │  │ Extraction│  │
```

Test Specification Format

Tests are defined as YAML files containing ordered sequences of steps. Each step specifies the commands to execute, expected output patterns for pass/fail evaluation, timeout duration, and optional variable extraction patterns. An Excel-to-YAML converter (excel_to_yaml.py) auto-detects Test ID patterns from column A and generates one YAML file per test case.

```yaml
name: "Validate resource lifecycle"
steps:
  - name: "Generate resource and extract ID"
    command: "bash"
    responses:
      - "admin-tool generate-resource"
    expected_output: "SUCCESS"
    wait: 5
    extract:
      resource_id: "RESOURCE-ID:\\s*(\\S+)"

  - name: "Delete resource using extracted ID"
    command: "bash"
    responses:
      - "admin-tool delete {{resource_id}}"
    expected_output: "SUCCESS"
    wait: 5
```

Dynamic Variable Chaining via Regex Extraction

Steps define `extract:` blocks containing named regex patterns applied against command output. Captured values are stored in a shared variable context and substituted into subsequent steps via {{variable}} template syntax. This mechanism enables multi-step test workflows where runtime-generated identifiers flow automatically between steps without manual intervention. Variables persist across all steps within a single test case, enabling chains such as: generate resource → extract ID → use ID to query → use ID to modify → use ID to delete.

Crash-Resilient Execution Engine

The SSH executor (test_runner.py) wraps each test case in isolated exception handling with configurable per-step timeouts. The Paramiko-based SSH connector establishes interactive sessions, where commands are written to stdin and output is collected via a background reader thread with timeout detection. If any step crashes, times out, or the SSH connection drops, the framework:
(1) Records the failure state with full output captured up to the point of failure,
(2) Closes the failed SSH session,
(3) Establishes a fresh connection for the next test case,
(4) Continues execution of remaining tests.

This isolation ensures that no single test failure can halt the pipeline.

Deterministic Failure Analysis

The analysis engine (pipeline.py) performs three levels of automated failure detection without involving any LLM:

Level 1 — Expected Output Matching: Each step's actual output is compared against its declared expected_output pattern.

Level 2 — Error Pattern Detection: Output lines are scanned against a configurable set of error keywords. Matches are recorded with line numbers and surrounding context (±3 lines).

Level 3 — Root-Cause Classification: Detected error patterns are mapped to diagnostic categories:

| Error Pattern                    | Root-Cause Classification                          |
|----------------------------------|----------------------------------------------------|
| "not found" / "not configured"   | Resource not configured or does not exist          |
| "denied" / "not authorized"      | Authentication or permission failure               |
| "syntax" / "illegal"             | Command not recognized or invalid format           |
| Timeout / no output              | Process timeout — no response within window        |
| "already" / "in use"             | Resource contention or state conflict              |

AI Agent Integration for Report Generation

Four AI agent tools defined in server/mcp_server.py and exposed via Model Context Protocol (MCP) server provide the intelligence layer:

| Tool                | Function                                              |
|---------------------|-------------------------------------------------------|
| run_tests           | Execute test cases over SSH                           |
| collect_logs        | Save per-step output to structured log files          |
| analyze_logs        | Pinpoint failure context with line numbers            |
| generate_report     | Produce human-readable reports from structured data   |

Critically, the AI agents receive only the pre-analyzed structured summary — not raw logs. The local pattern-matching engine reduces thousands of output lines into a compact JSON structure. This distilled payload (~1,500 tokens) is passed for formatting into a human-readable report with actionable recommendations. This architectural decision enables the 97% token reduction compared to the manual approach.
